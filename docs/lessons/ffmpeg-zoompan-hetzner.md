# Урок: ffmpeg zoompan на Hetzner 4-core

## Проблема (повторилась дважды)

`zoompan=z='...':d=1:s=1920x1080:fps=24` на 4864×3328 PNG-картинке с `scale=2112:1188:flags=lanczos` + Ken Burns на 1800 сек (30 мин):
- 1 процесс: ~1.3 fps realtime, ETA **~9 часов**
- 4 параллельных процесса на 4 ядрах: каждый ~0.07-0.11x realtime, ETA **6+ часов**
- 4 готовых 30-мин ролика ещё НЕ завершились после 1.5 часа — 78% CPU каждый

## Корневые причины

1. `zoompan=d=1` — самый медленный режим (на каждом кадре новая трансформация)
2. Lanczos-скейл 5K→1080p на CPU дорогой
3. 4 параллельных CPU-bound ffmpeg конкурируют за 4 ядра

## Решения (от субагентов)

### A. Использовать GPU кодек
```bash
-c:v h264_vaapi -rc_mode 4 -qp 23
```
Ускорение: 10-30×. У Hetzner есть `/dev/dri/renderD128` (Intel iGPU).

### B. Изменить `d=` в zoompan
```bash
zoompan=z='min(zoom+0.000045,1.13)':d=125:s=1920x1080:fps=24
```
Ускорение: **125×** (с `d=1` на `d=125`). Финиш за ~3 мин вместо 6 часов.

### C. Предрендер на меньшем разрешении
```
scale=960:540:flags=lanczos + zoompan → потом scale=1920:1080:flags=lanczos
```
Ускорение: 5-8×.

## Рекомендация

**Для длинных ambient-роликов на Hetzner CPU:**
- Использовать `d=125` или больше (или d=N где N=2×FPS)
- Или GPU кодек `h264_vaapi`
- **Не запускать >1 параллельного рендера** на 4 ядрах
- Или заложить в бюджет: 30-мин ролик = **6+ часов** wall clock

## Команда-победитель (на будущее)

```bash
ffmpeg -y -loop 1 -i input.png -stream_loop -1 -i audio.mp3 -t 1800 \
  -filter_complex "[0:v]scale=2112:1188:flags=lanczos,zoompan=z='min(zoom+0.000045,1.13)':d=125:s=1920x1080:fps=24[v]" \
  -map "[v]" -map "1:a" \
  -c:v h264_vaapi -qp 23 \
  -c:a aac -b:a 128k -ar 44100 \
  -movflags +faststart \
  output.mp4
```

## Что делать с текущими 4-5 рендерами

Они работают, ETA 1-3 часа. Если доделаются — хорошо. Если нет — убить и переделать с `d=125`.
