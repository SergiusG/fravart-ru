## 📅 23 июля 2026 — HappyHorse batch-квота Alibaba

### Что произошло

Попытались сгенерировать 28 видеоклипов для Buried Blueprints Episode 2 (VVA-14) через Alibaba Cloud HappyHorse 1.1 (новый «топ-1 в мире» генератор). Запустили batch-скрипт с retry-механизмом и экспоненциальной задержкой. Получили **0 готовых клипов** и сожгли **недельную квоту** Token Plan. Результат: до 30 июля видео через Alibaba недоступно.

### Корневые причины

1. **Alibaba Token Plan ($18/мес) — СНЕПШОТ-квота**, не per-minute rate limit. Квота обнуляется раз в неделю, не «остывает» после паузы.
2. **Скрипт стартовал без проверки формата лимита** — мы предполагали sliding rate limit, а оказался weekly snapshot.
3. **Поллинг каждые 10 секунд** на 28 клипов параллельно — сам по себе сжёг квоту быстрее, чем генерация.
4. **Retry с exponential backoff** только ухудшил ситуацию: вместо паузы и анализа — продолжали долбить API.

### Анти-паттерны

- ❌ Запускать batch без предварительного одиночного теста
- ❌ Сжигать ресурсы, не зная точно формат лимита
- ❌ Retry-логика без понимания причины 429
- ❌ Продолжать ту же стратегию после первого 429

### Правильный путь

1. **Перед любым batch — 1 одиночный тест.** Если 429 → остановиться, изучить формат лимита.
2. **Сначала определить лимит:** sliding rate limit (остывает) vs snapshot quota (нет).
3. **Если snapshot — посчитать:** 1 клип = X% квоты → N клипов = N×X%. Если > 50% — спросить пользователя.
4. **Если лимит сгорел — зафиксировать дату сброса**, не повторять попытки.
5. **Иметь запасной план** (другая модель/провайдер) ДО старта, а не после провала.

### Контекст

- API: `https://token-plan.ap-southeast-1.maas.aliyuncs.com/api/v1/services/aigc/video-generation/video-synthesis`
- Модель: `HappyHorse-1.1` (в Token Plan НЕ входит, требует DashScope с отдельной оплатой)
- Альтернативы: Kling MCP (~30 кредитов), MiniMax Hailuo, Veo

## 📅 23 июля 2026 — Зацикливание на мёртвой задаче

### Что произошло

После того как HappyHorse вернул 429, продолжали retry вместо того чтобы остановиться и спросить пользователя. Сожгли ещё несколько попыток перед тем, как понять природу лимита.

### Правило

**Если задача не движется 2 попытки — СТОП. Спросить пользователя.** Не тратить ресурсы впустую.

## 📅 Общий принцип: пользователь в офлайне

### Что произошло

Сергей в мотоциклетном путешествии, в Telegram отвечает редко. Тем не менее, агент продолжал выполнять действия, которые требовали подтверждения.

### Правило

Если пользователь явно не в сети (нет ответа > 30 минут) и задача требует значимых ресурсов — **остановиться после первого сбоя**, не выжигать квоту. Лучше оставить задачу на момент, когда пользователь сможет подтвердить.

## 💾 Ключевые выводы

1. **Перед batch — всегда 1 тест.**
2. **Перед retry — понять причину сбоя.**
3. **Перед сжиганием квоты — спросить пользователя.**
4. **Иметь альтернативу ДО старта, не после провала.**
5. **Каждый значимый кейс — записывать в вики** для памяти между сессиями.
